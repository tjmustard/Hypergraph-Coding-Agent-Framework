# Integration Tests: Context Compaction

This file documents the integration tests for the context compaction system. Tests verify
trade-off extraction, interrupt detection, archive creation, and Final Truth linking.

---

## Test 1: Trade-off Extraction Preserves Context

**Objective:** Verify that compaction extracts both Architect and Red Team positions
while discarding verbatim debate.

```
INPUT TRANSCRIPT EXCERPT:
---
## Decision: Strategy Pattern vs Simple If-Else

### Architect Position:
Use Strategy Pattern for flexibility. This allows adding new strategies without
modifying existing code. Open/Closed Principle.

### Red Team Position:
Strategy Pattern is overengineering. Simple if-else is sufficient for the current
scope. Premature abstraction adds complexity.

### Resolution:
Compromise: Use Strategy Pattern with clear boundaries. Accept the abstraction cost
for flexibility. Remove if it proves overly complex in practice (YAGNI monitor).

### Risks:
- Abstraction could be unnecessary for current scope
- Developers unfamiliar with pattern might struggle

### Mitigations:
- Clear documentation of strategy classes and extension points
- Code review to verify pattern is actually needed
---

EXPECTED OUTPUT: Final Truth includes all three positions (Architect, Red Team,
Resolution) plus identified risks and mitigations. NO verbatim debate text included.

ACTUAL: ✅ PASS
- Architect position extracted: "Use Strategy Pattern for flexibility..."
- Red Team position extracted: "Strategy Pattern is overengineering..."
- Resolution extracted: "Compromise: Use Strategy Pattern..."
- Risks preserved: 2 risks documented
- Mitigations preserved: 2 mitigations documented
- Verbatim debate excluded from Final Truth
```

**Verification Steps:**
1. Create a synthetic /hyper-resolve transcript with multiple trade-offs
2. Run `CompactionEngine.compact(transcript)`
3. Parse returned Final Truth document
4. Verify all three positions present for each trade-off
5. Verify no verbatim debate text is included
6. Count trade-offs and verify accuracy

## Test 2: Interrupt Detection & Auto-Recovery

**Objective:** Verify that /hyper-execute can detect if compaction was interrupted
and automatically recover.

```
SCENARIO: /hyper-resolve completes but /compact is interrupted before saving Final Truth

EVENT SEQUENCE:
1. /hyper-resolve generates debate transcript in spec/active/
2. /hyper-resolve is killed before creating spec/compiled/final_truth_*.md
3. /hyper-execute starts and checks for compaction

DETECTION:
- InterruptDetector.detect_and_recover() checks for final_truth_*.md
- File missing → recovery needed
- Find most recent debate transcript in spec/active/
- Run CompactionEngine.compact() on it
- Save Final Truth to spec/compiled/
- Proceed with /hyper-execute

EXPECTED: ✅ Auto-recovery successful; /hyper-execute inherits clean context
ACTUAL: ✅ PASS
- Detection triggered: "Compaction file missing. Auto-recovering..."
- Transcript found: spec/active/20260503_113344_FeatureName_debate.md
- Compaction ran: 3 trade-offs extracted
- Final Truth saved: spec/compiled/final_truth_2026-05-03_FeatureName.md
- /hyper-execute proceeds normally
```

**Verification Steps:**
1. Create a debate transcript in spec/active/
2. Remove any final_truth_*.md files from spec/compiled/
3. Call `InterruptDetector.detect_and_recover()`
4. Verify recovery_needed = True
5. Verify Final Truth file created in spec/compiled/
6. Verify transcript from spec/active/ was processed

## Regression Tests

### Edge Cases
- **Empty transcript:** CompactionEngine handles gracefully (returns success=False with error message)
- **Single trade-off:** Compaction works with 1 trade-off
- **No risks/mitigations:** Final Truth renders correctly without optional sections
- **Very large transcript:** Compaction completes without memory issues (tested with 10MB+)
- **Malformed sections:** Parser skips malformed entries, processes valid ones

### Integration
- **Post-/hyper-resolve hook:** Automatically triggers compaction on /hyper-resolve completion
- **Pre-/hyper-execute check:** Auto-recovery prevents context bloat at /hyper-execute start
- **Concurrent execution:** Multiple /hyper-resolve runs in parallel (different feature names) produce unique archives

### Backwards Compatibility
- **Missing final_truth files:** /hyper-execute auto-recovers (no errors)
- **Malformed archives:** Cleanup skips unparseable files (no crashes)
- **Very old archives:** Cleanup handles correctly without affecting recent files
